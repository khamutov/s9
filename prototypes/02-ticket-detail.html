Crash on startup when config is missing

S9-42
Bug In Progress P1
MC Maria Chen opened 3 days ago

The application panics on startup when the config.toml file is missing from the expected path. Instead of falling back to defaults or showing a helpful error, we get an unhandled unwrap() on None.

Steps to reproduce:

  1. Delete or rename ~/.config/s9/config.toml
  2. Run s9 serve
  3. Observe panic with backtrace

Expected behavior:

The server should start with sensible defaults and log a warning about the missing config file. See #S9-38 for the related config refactor discussion.

Stack trace:

thread 'main' panicked at 'called `Option::unwrap()`
on a `None` value', src/config/loader.rs:42:18
stack backtrace:
   0: rust_begin_unwind
   1: core::panicking::panic
   2: s9::config::loader::load_config
   3: s9::main

This blocks #S9-45 (deployment automation) since the CI environment doesn't have a pre-provisioned config file.

AK Alex Kim #1 2 days ago

I can reproduce this consistently. The issue is in loader.rs line 42 — we call .unwrap() directly on the File::open result instead of handling the NotFound case.

Proposed fix: fall back to Config::default() when the file doesn't exist, and only propagate errors for permission issues or malformed TOML. Something like:

let config = match File::open(&config_path) {
    Ok(file) => parse_config(file)?,
    Err(e) if e.kind() == NotFound => {
        warn!("Config not found at {}, using defaults", config_path);
        Config::default()
    }
    Err(e) => return Err(e.into()),
};

@Maria Chen does that approach work for you, or do you want a stricter "require config in production" mode?

MC Maria Chen #2 2 days ago

That approach looks good. Let's keep it simple — defaults for dev, and we can add a --require-config flag later if needed for production. The Config::default() values should match what we document in the README.

One thing to watch out for: the DNS resolver config in /Platform/Networking/DNS/ has its own config section. Make sure the defaults there point to the system resolver, not our custom one.

Linking #S9-38 as related.

BL Bob Lee #3 1 day ago

FYI — I hit this same issue on the staging deploy yesterday. Adding a note that the current workaround is to manually create an empty config:

mkdir -p ~/.config/s9
touch ~/.config/s9/config.toml

@Alex Kim once your fix is in, we should also add a config.toml.example to the repo root so people have a starting point. @Jess Wong can you update the deployment docs in #S9-50 once this lands?
