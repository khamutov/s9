## 1. Purpose

Tickets currently display as `#23` — a bare numeric ID that carries no context about which part of the system it belongs to. Ticket slugs add a component-derived prefix so tickets are referenced as `#MAP-23` or `#PLAT-107`. This makes ticket references instantly recognizable in conversation, commit messages, and the filter micro-syntax.

## 2. Component Slugs

Each component may define an optional **slug** — a short uppercase identifier (e.g. `MAP`, `PLAT`, `NET`).

### 2.1 Rules

| Rule | Detail |
|------|--------|
| Format | 2–10 uppercase ASCII characters. Must start with a letter. Digits allowed after the first character. |
| Regex | `^[A-Z][A-Z0-9]{1,9}$` |
| Uniqueness | Globally unique across all components. No two components may share a slug. |
| Optionality | Slugs are optional on non-root components. **Root components must have a slug** (enforced at creation and update time). |

### 2.2 Inheritance

A component's **effective slug** is resolved by walking up the component tree from the component itself toward the root, returning the first non-null `slug` encountered. Because root components always have a slug, resolution is guaranteed to succeed.

Examples:

```
Platform (slug: PLAT)
├── Networking (slug: NET)
│   └── DNS (no slug)           → effective slug: NET
├── Storage (no slug)           → effective slug: PLAT
│   └── S3 Adapter (slug: S3A) → effective slug: S3A
└── Auth (no slug)              → effective slug: PLAT
```

### 2.3 Display

When listing or displaying a component, show the effective slug as a badge or label alongside the component name. Components that define their own slug show it directly; components that inherit show the inherited slug in a muted style.

## 3. Ticket Slugs

A ticket's slug is **computed, not stored**. It combines the effective slug of the ticket's component with the ticket's numeric ID.

| Element | Source | Example |
|---------|--------|---------|
| Prefix | Effective slug of the ticket's component | `MAP` |
| Separator | Hyphen `-` | `-` |
| Number | Ticket's global auto-increment ID | `23` |
| Full slug | Concatenation | `MAP-23` |

Ticket slugs are displayed wherever a ticket ID appears: lists, detail views, comments, and notifications.

### 3.1 Slug stability

When a ticket moves to a different component, or a component's slug changes, the ticket's displayed slug changes accordingly. The ticket's numeric ID never changes. References using the old slug will no longer resolve — this is acceptable given the administrative nature of slug changes.

## 4. Micro-syntax Changes

### 4.1 Ticket references

The existing `#ID` reference syntax is extended to support slug-prefixed references.

| Syntax | Pattern | Example | Resolution |
|--------|---------|---------|------------|
| Numeric | `#\d+` | `#23` | Look up ticket by ID directly. |
| Slug-prefixed | `#[A-Z][A-Z0-9]+-\d+` | `#MAP-23` | Parse prefix and ID. Verify the ticket exists and its effective slug matches the prefix. |

Both forms are valid in comments, descriptions, and anywhere micro-syntax is parsed. The slug-prefixed form takes parsing priority (longer match).

### 4.2 Search filter syntax

A new filter is added:

| Filter | Example | Description |
|--------|---------|-------------|
| `slug:prefix` | `slug:MAP` | Tickets whose effective component slug matches the given prefix. |

This complements the existing `component:path` filter with a shorter alternative.
